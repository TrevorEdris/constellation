# Terraform Module Structure

## Standard Layout

Every module ships these files. No monolith `.tf` files.

```
my-module/
├── main.tf          # Primary resources
├── variables.tf     # All variable declarations
├── outputs.tf       # All output declarations
├── versions.tf      # terraform {} block
└── README.md        # Generated by terraform-docs
```

When `main.tf` exceeds ~150 lines, extract by concern:

```
my-module/
├── main.tf          # Entry point and shared resources
├── iam.tf           # IAM roles, policies, attachments
├── networking.tf    # VPCs, subnets, security groups
├── storage.tf       # S3, EFS, RDS
├── variables.tf
├── outputs.tf
└── versions.tf
```

## Root Module vs Child Module

| Aspect | Root Module | Child Module |
|--------|-------------|--------------|
| Purpose | Deployable unit | Reusable library |
| Backend config | Yes (`backend {}` in `versions.tf`) | No backend block |
| Provider config | Yes (full `provider {}` block) | No — receive from root via `providers` argument |
| State | Has its own state file | No state |
| Invoked by | CLI (`terraform apply`) | Root modules via `module {}` block |

**Never** mix root and child module concerns in the same directory.

## `versions.tf` Template

```hcl
terraform {
  required_version = ">= 1.6.0"

  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.0"
    }
  }

  backend "s3" {
    # Values injected via -backend-config or wrapper tooling
    # Keeping this block empty is valid for partial configuration
  }
}
```

## Module Versioning

Always pin child modules to a specific semver tag. Never `?ref=main` or `?ref=HEAD`.

```hcl
# Correct — pinned to exact tag
module "vpc" {
  source  = "terraform-aws-modules/vpc/aws"
  version = "5.5.3"
}

# Correct — git source pinned to tag
module "my_module" {
  source = "git::https://github.com/org/terraform-aws-mymodule.git?ref=v1.4.2"
}

# Wrong — unpinned, breaking changes flow to all callers immediately
module "vpc" {
  source = "terraform-aws-modules/vpc/aws"
  # no version constraint!
}
```

## Module Responsibility

One module = one responsibility. Multi-purpose "platform" modules become unmaintainable:
- Networking: VPCs, subnets, route tables, NAT gateways, peering
- Compute: EC2, ECS, EKS, Lambda
- Storage: S3, RDS, ElastiCache
- IAM: Roles, policies, instance profiles

## Terraform Registry Publishing

Public registry modules require:
- Repo name: `terraform-<provider>-<name>` (e.g., `terraform-aws-vpc`)
- GitHub releases with semver tags
- `LICENSE` file
- Structured module layout (the registry auto-discovers `variables.tf`, `outputs.tf`)

## `terraform-docs` Integration

Generate README inputs/outputs table:

```bash
# Install
brew install terraform-docs

# Generate and write to README.md
terraform-docs markdown . > README.md

# Or inject between markers in existing README
terraform-docs markdown --output-file README.md .
```

Add to `.pre-commit-config.yaml`:

```yaml
- repo: https://github.com/terraform-docs/terraform-docs
  rev: "v0.18.0"
  hooks:
    - id: terraform-docs-go
      args: ["markdown", ".", "--output-file", "README.md"]
```
