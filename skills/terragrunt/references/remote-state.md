# Terragrunt Remote State

## `remote_state` Block

Declare once in the root `terragrunt.hcl`. Terragrunt auto-generates `backend.tf` in each child unit, eliminating the most common source of copy-paste across Terraform modules.

```hcl
# Root terragrunt.hcl
remote_state {
  backend = "s3"

  # Controls the generated backend.tf
  generate = {
    path      = "backend.tf"
    if_exists = "overwrite_terragrunt"
  }

  config = {
    bucket         = "myorg-tfstate-${get_aws_account_id()}"
    key            = "${path_relative_to_include()}/terraform.tfstate"
    region         = "us-east-1"
    encrypt        = true
    kms_key_id     = "arn:aws:kms:us-east-1:${get_aws_account_id()}:alias/terraform-state"
    dynamodb_table = "terraform-state-lock"

    # S3 access logging
    accesslogging_bucket_name = "myorg-tfstate-access-logs"
  }
}
```

## Path-Based State Keys

Use `path_relative_to_include()` as the state key. This ensures:
- Unique state path per unit
- Human-readable path matching directory structure
- Browsable state bucket that mirrors your repo layout

```
s3://myorg-tfstate-123456789012/
├── prod/
│   ├── us-east-1/
│   │   ├── networking/terraform.tfstate
│   │   ├── api/terraform.tfstate
│   │   └── database/terraform.tfstate
│   └── eu-west-1/
│       └── networking/terraform.tfstate
└── staging/
    └── us-east-1/
        └── networking/terraform.tfstate
```

## GCS Remote State

```hcl
remote_state {
  backend = "gcs"
  generate = {
    path      = "backend.tf"
    if_exists = "overwrite_terragrunt"
  }
  config = {
    project  = "myorg-terraform"
    location = "us-central1"
    bucket   = "myorg-tfstate"
    prefix   = path_relative_to_include()
  }
}
```

## Azure Blob Remote State

```hcl
remote_state {
  backend = "azurerm"
  generate = {
    path      = "backend.tf"
    if_exists = "overwrite_terragrunt"
  }
  config = {
    resource_group_name  = "terraform-state-rg"
    storage_account_name = "myorgtfstate"
    container_name       = "tfstate"
    key                  = "${path_relative_to_include()}/terraform.tfstate"
  }
}
```

## Auto-Create Backend Resources

Terragrunt can provision the S3 bucket and DynamoDB table on first use if they don't exist:

```hcl
remote_state {
  backend = "s3"
  generate = {
    path      = "backend.tf"
    if_exists = "overwrite_terragrunt"
  }
  config = {
    bucket         = "myorg-tfstate"
    key            = "${path_relative_to_include()}/terraform.tfstate"
    region         = "us-east-1"
    encrypt        = true
    dynamodb_table = "terraform-state-lock"

    # Auto-create resources if missing
    s3_bucket_tags = {
      Name        = "Terraform State"
      ManagedBy   = "terragrunt"
    }
    dynamodb_table_tags = {
      Name      = "Terraform State Lock"
      ManagedBy = "terragrunt"
    }
  }
}
```

**Warning:** Auto-creation requires the IAM role to have S3 and DynamoDB create permissions. In hardened environments, pre-create the backend resources via a bootstrap module and remove auto-create permissions from the Terragrunt role.

## Accessing Remote State from a Dependency

Child units access other units' state through `dependency` blocks — not by configuring `remote_state` directly. See `dependency-management.md` for the full pattern.

## Generated `backend.tf` Example

When Terragrunt runs `init` in a child unit, it writes a `backend.tf` like:

```hcl
# DO NOT EDIT: generated by Terragrunt
terraform {
  backend "s3" {
    bucket         = "myorg-tfstate-123456789012"
    key            = "prod/us-east-1/api/terraform.tfstate"
    region         = "us-east-1"
    encrypt        = true
    dynamodb_table = "terraform-state-lock"
  }
}
```

This file should be in `.gitignore` — it is generated at init time and varies per environment.

```gitignore
# Terragrunt generated files
**/.terraform/
**/.terragrunt-cache/
**/backend.tf
**/provider.tf
```
